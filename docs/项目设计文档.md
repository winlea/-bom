# BOM 表图像管理系统 - 项目设计文档

## 1. 项目概述

### 1.1 项目背景
BOM 表图像管理系统是一个基于 GD&T（几何尺寸与公差）标准的尺寸添加及管理系统，旨在为制造企业提供高效、规范的尺寸管理解决方案。

### 1.2 核心价值
- **省时**：比传统方式节省 50% 的时间
- **省心**：一站式解决方案，无需切换多个工具
- **省力**：简洁直观，低学习成本

### 1.3 技术栈
- **后端**：Flask + SQLite + SQLAlchemy
- **前端**：React 18+ + TypeScript + Tailwind CSS
- **状态管理**：Zustand
- **API通信**：Axios
- **开发工具**：Vite

## 2. 系统架构设计

### 2.1 整体架构
系统采用前后端分离架构，通过 RESTful API 进行通信。

```
┌─────────────────┐
│   前端应用      │
│  (React/TS)    │
└────────┬────────┘
         │ HTTP/REST
         │
┌────────┴────────┐
│   后端服务      │
│   (Flask)       │
└────────┬────────┘
         │
┌────────┴────────┐
│   SQLite 数据库 │
└─────────────────┘
```

### 2.2 后端架构

#### 2.2.1 核心组件
- **服务接口层** (`interfaces/`)：定义业务逻辑接口
- **仓储实现层** (`repositories/`)：实现数据持久化
- **依赖注入容器** (`container.py`)：管理依赖关系
- **数据库会话管理** (`database/session.py`)：统一管理数据库连接
- **配置管理** (`config/manager.py`)：环境配置管理
- **标准化API响应** (`api/response.py`)：统一响应格式
- **错误处理** (`api/error_handler.py`)：全局异常处理

#### 2.2.2 设计模式
- **依赖注入模式**：通过 Dependency Injector 实现松耦合
- **仓储模式**：数据访问与业务逻辑分离
- **服务层模式**：封装核心业务逻辑

## 4. 核心功能设计

### 4.1 尺寸管理功能

#### 4.1.1 尺寸类型系统
支持以下尺寸类型：
- **普通尺寸**：线性尺寸标注
- **孔径 ⌀**：直径尺寸标注
- **半径 R**：半径尺寸标注
- **球直径 S⌀**：球直径尺寸标注
- **球半径 SR**：球半径尺寸标注

几何公差类型：
- **形状公差**：直线度 ⏤、平面度 ⏥、圆度 ○、圆柱度 ⌭
- **轮廓公差**：线轮廓 ⌒、面轮廓 ⌓
- **方向公差**：平行度 ∥、垂直度 ⊥、倾斜度 ∠
- **位置公差**：位置度 ⌖、同心度 ◎、对称度 ⌯
- **跳动公差**：圆跳动 ↗、全跳动 ↗↗
- **其他**：共面度 ⏥⏥

#### 4.1.2 基准系统
- 支持单个基准（如：A、B1）
- 支持基准体系（如：A-B-C、A1-B2-C3）
- 基准使用 `-` 分隔，符合 ISO 1101/ASME Y14.5 和 GB/T 1182 标准

#### 4.1.3 尺寸组合逻辑
**规则**：
- 一个尺寸组合（group）中只能有一个孔径大小（diameter）
- 一个孔径大小可以对应一个或多个位置度（position）
- 位置度必须指定基准
- 基准符号格式必须符合标准要求

**实现方式**：
- 在添加新尺寸时进行验证，防止重复添加孔径尺寸
- 在编辑尺寸时进行验证，确保编辑后不会违反尺寸组合规则
- 在批量添加尺寸时进行验证，确保批量添加的尺寸组合合理

### 4.2 尺寸图片生成

#### 4.2.1 普通尺寸绘制
- 显示尺寸符号（如：⌀、R）
- 显示名义值
- 显示公差（对称/不对称/极限）
- 使用 Canvas 绘制，支持自定义字体大小

#### 4.2.2 几何公差绘制
- 几何公差框布局：符号框 → 公差框 → 基准框
- 支持动态调整框大小以适应内容
- 基准框自动调整宽度以适应包含数字的基准符号

#### 4.2.3 尺寸图片预览
- 点击尺寸图片可以放大预览
- 预览图片使用更大的画布和字体
- 支持图片下载功能

### 4.3 组合图片生成
- 支持将同一组的多个尺寸组合成一张图片
- 自动排版尺寸标注
- 支持组号管理

### 4.4 尺寸识别功能
- 支持从图片中识别尺寸
- 识别结果可编辑和删除
- 批量导入识别结果

## 5. API 设计

### 5.1 标准化响应格式
```json
{
  "success": true,
  "message": "Success",
  "data": {
    "items": [...],
    "total": 100
  }
}
```

### 5.2 核心 API 端点

#### 5.2.1 项目管理
- `GET /api/projects` - 获取项目列表
- `POST /api/projects` - 创建项目
- `PUT /api/projects/:id` - 更新项目
- `DELETE /api/projects/:id` - 删除项目

#### 5.2.2 零件管理
- `GET /api/parts?project_id=:id` - 获取零件列表
- `POST /api/parts` - 创建零件
- `PUT /api/parts/:id` - 更新零件
- `DELETE /api/parts/:id` - 删除零件

#### 5.2.3 尺寸管理
- `GET /api/dimensions/projects/:projectId?part_number=:partNumber` - 获取尺寸列表
- `POST /api/dimensions/projects/:projectId` - 创建尺寸
- `PUT /api/dimensions/:id` - 更新尺寸
- `DELETE /api/dimensions/:id` - 删除尺寸
- `POST /api/dimensions/projects/:projectId/bulk` - 批量创建尺寸
- `POST /api/dimensions/projects/:projectId/insert` - 插入尺寸
- `POST /api/dimensions/projects/:projectId/image-dimension` - 创建图片尺寸
- `GET /api/dimensions/export/excel` - 导出Excel

## 6. 前端实现思路

### 6.1 状态管理
使用 Zustand 进行全局状态管理：
- 项目列表状态
- 当前选中项目状态
- 零件列表状态
- 当前选中零件状态
- 尺寸列表状态
- 尺寸预览模态框状态

### 6.2 数据获取流程
```
用户选择项目
    ↓
获取项目零件列表
    ↓
用户选择零件
    ↓
获取零件尺寸列表
    ↓
渲染尺寸表格
```

### 6.3 组件设计原则
- **单一职责**：每个组件只负责一个功能
- **可复用性**：基础UI组件可在多个场景复用
- **可测试性**：组件逻辑与渲染分离
- **性能优化**：使用 useMemo、useCallback 优化重渲染

### 6.4 错误处理
- API 请求错误统一处理
- 用户友好的错误提示
- 网络超时重试机制
- 表单验证实时反馈
